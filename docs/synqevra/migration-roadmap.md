# Synqevra — Migration Roadmap (Этап 1)

## Общая информация

**Цель Этапа 1:** Полностью убрать зависимость от ThingsBoard.

**Подход:** Strangler Fig Pattern + постепенная замена через существующий слой адаптеров.

**Ориентировочная длительность:** 9–14 месяцев

---

## Phase 0: Подготовка и проектирование (3–5 недель)

**Цели:**
- Полная инвентаризация использования ThingsBoard
- Проектирование новой доменной модели
- Выбор технологического стека
- Подготовка инфраструктуры и feature flags

**Ключевые результаты:**
- Документ "ThingsBoard Dependency Map"
- Утверждённая доменная модель
- PoC по time-series хранилищу и MQTT
- Настроенные feature flags

---

## Phase 1: Реестр сущностей (Entities) (5–7 недель)

**Цель:** Наш бэкенд становится источником правды для устройств, ассетов, профилей и тенантов.

**Основные задачи:**
- Создать внутренние сервисы для Device, Asset, Tenant, Profiles
- Реализовать CRUD
- Миграция данных
- Dual-write на переходный период
- Обновление связей с Medplum

**Acceptance Criteria:**
- Все CRUD операции по устройствам и ассетам работают без обращения к ThingsBoard

---

## Phase 2: Телеметрия и атрибуты (6–8 недель)

**Цель:** Полный контроль над хранением и доступом к телеметрии.

**Ключевые работы:**
- Внедрение TimescaleDB / ClickHouse
- Реализация ingestion (HTTP + MQTT)
- API для latest/history/attributes
- Shadow-write телеметрии
- Перевод всех чтений телеметрии на новое хранилище

---

## Phase 3: Обработка данных и Alarms (6–8 недель)

**Цель:** Забрать бизнес-логику из ThingsBoard Rule Engine.

**Важно:**
- Перенести логику из `base_rule_chain.json`
- Развитие существующего домена Alarm + FSM
- Прямая обработка в ingestion pipeline
- Улучшение outbox и доставки алертов

**Acceptance Criteria:**
- Логика проверки порогов и генерации алертов работает без ThingsBoard

---

## Phase 4: Дашборды и визуализация (8–12 недель)

**Самая сложная фаза.**

**Задачи:**
- Принять стратегию визуализации (свой builder + опционально Grafana)
- Модель Dashboard + Widget
- Runtime рендеринга
- Убрать iframe-прокси (порт 3002)
- Live-обновления через WebSocket

---

## Phase 5: Rule Engine и автоматизации (6–9 недель)

- Довести до ума кастомный редактор на React Flow
- Execution engine
- Миграция существующих rule chains

---

## Phase 6: Device Connectivity (5–7 недель)

- Собственная система credentials
- Поддержка MQTT и других протоколов
- OTA packages

---

## Phase 7–8: Cutover и Decommission (5–7 недель)

- Переключение feature flags
- Финальная миграция данных
- Полное удаление кода ThingsBoard
- Вывод ThingsBoard из эксплуатации

---

## Основные Milestones

- **M0** — Завершена подготовка
- **M1** — Сущности работают без TB
- **M2** — Телеметрия + алерты на своём стеке
- **M3** — Дашборды без iframe
- **M4** — Полный cutover
- **M5** — ThingsBoard decommissioned

---

## Риски (основные)

- Сложность миграции дашбордов и визуализации
- Производительность телеметрии
- Обратная совместимость существующих устройств
- Качество миграции rule logic
